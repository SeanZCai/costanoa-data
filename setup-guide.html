Send your Granola meeting notes to the team knowledge base

One-time setup, then your meeting notes upload automatically every morning. Notes you sync are attributed to you — so queries like "all the cybersecurity founders Greg has met" actually work.

What this does

Every Granola meeting note you take ends up in a shared Supabase database, structured into companies, individuals, and meetings tables. Future agents query that database to answer questions across the whole team's coverage. Each row is tagged with the teammate who synced it, so we can slice the data by VC.

Setup is now two commands and one paste — the plugin handles everything else.

Before you start

  • Claude Code installed (brew install --cask claude)
  • Your Granola account connected as an integration in Claude (Settings → Integrations → Granola)
  • Python 3.10+ on your laptop (python3 --version should show 3.10 or higher)
  • Your Supabase keys from Sean (sent via 1Password — don't put them in Slack)

One-time setup

1 Install the plugin ~30 sec

In Claude Code, run these two slash commands:

/plugin marketplace add SeanZCai/costanoa-data
/plugin install costanoa-data@costanoa-data

This pulls down the granola-sync and team-onboard skills plus the Python sync script. The repo is private, so Claude Code will use your GitHub auth.

2 Verify Granola is connected ~30 sec

In Claude Code, ask:

who am I signed in as on Granola?

Should return your @costanoa.vc email. If not, connect Granola in Settings → Integrations.

3 Run your first sync — paste keys when asked ~3 min

In Claude Code:

/granola-sync

The first run detects you haven't been set up yet and prompts you for:

  • SUPABASE_URL — paste it from the 1Password entry Sean shared.
  • SUPABASE_SERVICE_ROLE_KEY — same source.

The plugin writes both to ~/.costanoa-data/.env (chmod 600), creates a local venv at ~/.costanoa-data/.venv, installs the deps, and proceeds with the sync. Subsequent runs skip all that and just sync.

The service role key bypasses RLS. Once pasted, it lives only in ~/.costanoa-data/.env on your laptop — never committed, never logged. If your laptop is compromised, ask Sean to rotate the key in Supabase.
Heads up: the first sync is the slow one — it pulls all of your meetings in the last 30 days. Daily runs after that only fetch net-new and take well under a minute. To backfill further, say "sync the last 6 months".

Make it daily

Pick one. You don't need both.

Option A — /schedulerecommended

Runs in the cloud. Your laptop doesn't need to be on.

In Claude Code:

/schedule create daily at 7am: /granola-sync

Manage with /schedule list and /schedule delete <id>.

Option B — launchd (macOS)

Runs locally each morning. Use this if you'd rather not keep Claude Code involved.

cp automation/launchd/vc.costanoa.granola-sync.plist \
   ~/Library/LaunchAgents/
launchctl load \
   ~/Library/LaunchAgents/vc.costanoa.granola-sync.plist

Fires daily at 7am local. Logs to automation/launchd/sync.log. To stop, run launchctl unload on the same path.

How to verify it's working

  1. After the first sync, open the Supabase table editor and look at meetings. You should see your meetings, with created_by_team_member_id pointing at your row in team_members.
  2. Run /granola-sync a second time immediately — it should report "0 new meetings". That's idempotency working.
  3. Spot-check a meeting's summary_md against what you see in Granola. Should match.

What gets sent

  • Meeting title, date, AI-generated summary, and full verbatim transcript
  • Attendees (name + email) and their inferred company affiliations
  • For 1-on-1s where Granola only captured you, a low-confidence inference of the counterparty from the meeting title (so "Jigsaw <> Sean" becomes a queryable link to Jigsaw)

Meetings titled "Busy" or starting with "Busy " are skipped — they're calendar placeholders, not real meetings.

Joining the team roster

Your first /granola-sync automatically registers you in the team's team_members table — as long as your Granola account is on a @costanoa.vc or @costanoavc.com email. Future "all founders X has met" queries find you from then on.

If you have two Costanoa emails (e.g. one on each domain), they'll merge into a single profile by matching your full name. Just make sure your Granola account name matches what's in the team roster.

Helping a new hire onboard? Use /team-onboard newhire@costanoa.vc "Full Name" in Claude Code to register them before they themselves have set anything up.

Keeping certain meetings out of the knowledge base

Some meetings shouldn't end up in the shared DB — confidential board sessions, therapy, performance reviews, or just internal projects you don't want indexed. The sync supports a phrase-based exclusion filter that case-insensitively matches against the meeting's title, summary, and transcript. If any configured phrase appears anywhere in those three fields, that meeting is silently skipped.

For your daily automated uploads (set once)

Edit .env and set EXCLUDE_PHRASES to a comma-separated list. This applies to every sync — manual, /schedule cloud cron, and the launchd fallback. 

# In .env
EXCLUDE_PHRASES=board meeting,confidential,personal therapy,perf review

No quotes, no spaces around the commas. Leave it empty (EXCLUDE_PHRASES=) for no defaults.

For a one-off manual run

Just say it inline when invoking the skill: 

/granola-sync excluding "standard data", "investor diligence"

These get merged with your .env defaults — the union applies. The skill drops matching meetings before they're even fetched from Granola (saves API calls), and sync.py re-checks at insertion time as a defense-in-depth backstop.

How aggressive is the match? Case-insensitive substring. EXCLUDE_PHRASES=acme drops a meeting titled "Acme Inc <> Sean" but also any meeting where the summary mentions "acme" in passing. Pick phrases specific enough to avoid false positives — acme followup or acme deep dive over just acme.

The skip is silent in the DB — excluded meetings don't show up anywhere. The sync output JSON does log which IDs were excluded and which phrase matched, so you can verify after a run.

Common issues

"Could not find the table 'public.meetings'"

The schema hasn't been applied to Supabase yet. Sean does this once for the team — ping him if you see this.

"Email ... is not a Costanoa domain"

Your Granola account is signed in with a non-Costanoa email. Switch to your @costanoa.vc account in Granola and re-run.

"who am I signed in as on Granola?" returns nothing

The Granola integration isn't connected in Claude. Open Claude Code → Settings → Integrations → Granola → Connect.

The launchd job doesn't fire

Check which claude on your laptop and compare to the path in the plist. Default is /opt/homebrew/bin/claude (Apple Silicon). Update ProgramArguments in the plist if your binary lives elsewhere.
